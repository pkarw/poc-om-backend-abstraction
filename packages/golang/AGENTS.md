# AGENTS.md — packages/golang

Agent guide for the Go port of Open Mercato backend modules. Read the repo-root
`AGENTS.md` and `specs/` first; this file maps upstream concepts to Go idioms.

## Stack

Go 1.23 · chi v5 (HTTP) · pgx/v5 + pgxpool (PostgreSQL) · golang-migrate v4
(SQL migrations) · go-redis v9 (Redis) · go-playground/validator v10
(validation) · godotenv (.env) · stdlib `testing`. Details and rationale:
[docs/stack.md](docs/stack.md). `go.sum` is generated by `go mod tidy`
(`make tidy`); the Dockerfile runs it during build.

## Layout

```
cmd/api/                 HTTP host: /healthz + mounts module routes under /api
cmd/worker/              queue worker host (one consumer loop per queue)
cmd/migrate/             applies migrations/ then exits
internal/platform/       framework runtime (upstream's cross-cutting packages)
  config/                env loading + validation (upstream env var names)
  db/                    pgxpool connect + golang-migrate runner
  redisconn/             Redis client factory
  queue/                 JobQueue interface, redis (BullMQ-style) + local impls
  events/                in-process event bus
  validation/            shared validator (Zod equivalent)
  registry/              Module type, Deps DI container, mount/subscribe helpers
internal/modules/        one Go package per Open Mercato module
  modules.go             explicit module registration (the "generated registry")
  healthcheck/           reference module — copy this pattern
migrations/              flat golang-migrate dir: NNNN_<module>_<desc>.{up,down}.sql
docs/decisions/          ADRs
```

## Conventions

Naming: upstream module ids stay snake_case (`health_check`) in `ModuleID`,
URLs, queue names, event names and table names. Go package names collapse the
underscore (`healthcheck`). JSON fields keep upstream camelCase/lowercase via
struct tags — never rely on Go field names. Exported Go identifiers are
PascalCase; handler factories are unexported (`handleGetHealthCheck`).

| Open Mercato (TS) concept | This package |
|---|---|
| `modules/<m>/api/<method>/<path>.ts` route file | handler in `internal/modules/<m>/api.go`, registered in `Module()` via `r.Get("/<path>", ...)` on the `/api` subrouter |
| `modules/<m>/data/entities.ts` (MikroORM) | plain structs in `internal/modules/<m>/entities.go` + hand-written SQL via pgx |
| `modules/<m>/data/validators.ts` (Zod) | structs with `validate` tags in `internal/modules/<m>/validators.go` + `validation.Struct()` |
| `modules/<m>/subscribers/*.ts` | `registry.Subscriber{Event, Handler}` entries in `Module()`, handlers in `subscribers.go` |
| `modules/<m>/workers/*.ts` | `registry.Worker{Queue, JobName, Handler}` entries in `Module()`, handlers in `worker.go` |
| `modules/<m>/acl.ts` feature ids | `Features []string` on `registry.Module` |
| `modules/<m>/di.ts` (Awilix) | `registry.Deps` struct passed to `Module(deps)`; modules close over it |
| MikroORM migrations | SQL files in `migrations/` (golang-migrate) |
| generated module registry | `internal/modules/modules.go` `All()` |
| `packages/queue` strategies | `internal/platform/queue` (`QUEUE_STRATEGY=local|redis`) |
| `packages/events` | `internal/platform/events.Bus` (in-process; distributed = porting task) |

## Module Porting Rules

1. Create `internal/modules/<module_id>/` (package name without underscores)
   with `module.go`, `api.go`, `entities.go`, `validators.go`, and
   `worker.go`/`subscribers.go` when upstream has them.
2. `module.go` exports `Module(deps *registry.Deps) registry.Module` plus
   `ModuleID`, queue-name and event-name constants — copied **verbatim** from
   upstream (snake_case ids, dotted event names).
3. Register the module with one line in `internal/modules/modules.go`.
4. Add migrations: next free `NNNN` prefix, `NNNN_<module_id>_<desc>.up.sql`
   + matching `.down.sql`, DDL translated from upstream's MikroORM
   entities/migrations with identical table/column names.
5. Translate each upstream route file into a handler; keep Zod semantics in
   `validate` tags (Zod `.min(1)` → `min=1`, `.email()` → `email`,
   `.optional()` → `omitempty`).
6. Add table-driven tests (`*_test.go` beside the code, `httptest` for
   handlers) and run `make test`.
7. Update root `MODULES.md` and record any idiomatic deviation as an ADR.

## API Compatibility Rules

- Paths, methods, status codes and JSON body shapes are byte-compatible with
  upstream: same `/api/<module>/...` paths, same field names and casing
  (enforce with `json:"..."` tags on every exported field), same error
  envelopes.
- Always encode from structs, never `map[string]any` — Go maps randomize key
  order.
- `GET /healthz` is liveness only (no db/redis) and returns
  `{"status":"ok","service":"golang-api"}`.
- Auth semantics (JWT via `JWT_SECRET`) must match upstream once the `auth`
  module is ported; until then do not invent interim schemes.

## Data Layer

PostgreSQL via pgx/v5 (`pgxpool`), migrations via golang-migrate
([ADR 0002](docs/decisions/0002-pgx-and-golang-migrate.md)). Schema-parity
rules: table and column names, types, nullability, defaults, indexes and FK
names identical to upstream's generated schema — verify by diffing against
upstream's MikroORM migrations (`packages/core/src/modules/<m>/…` +
migration output). Adding a migration: create both `.up.sql` and `.down.sql`
in `migrations/`, then `make migrate` (natively) — the api container applies
them automatically on start. Never edit an applied migration; add a new one.

## Queues & Events

Redis + `internal/platform/queue.JobQueue` (`Enqueue`/`Process`), strategy
from `QUEUE_STRATEGY` (`redis` default, `local` in-process for dev/tests).
The redis strategy uses BullMQ key naming (`bull:<queue>:wait|active|...`,
job hashes) but is **not yet wire-compatible** with Node BullMQ — no official
Go client exists; the plan and honest status live in
[ADR 0003](docs/decisions/0003-bullmq-queue-compatibility.md). Queue names and
job names must match upstream exactly. Events: in-process `events.Bus`;
module subscribers auto-wire via the registry. Cross-process event delivery
(upstream redis events strategy) is an open porting task.

## Configuration

Loaded by `internal/platform/config` from env, with `.env` support (see
`.env.example`). Names are upstream-identical:

| Var | Required | Default | Meaning |
|---|---|---|---|
| `DATABASE_URL` | yes | — | Postgres DSN (`postgres://…?sslmode=disable` for dev) |
| `REDIS_URL` | yes | — | Redis for cache/general use |
| `QUEUE_STRATEGY` | no | `redis` | `local` or `redis` |
| `QUEUE_REDIS_URL` | no | `REDIS_URL` | dedicated queue Redis |
| `JWT_SECRET` | no (yet) | — | auth signing secret (used once auth is ported) |
| `PORT` | no | `8090` | API listen port |
| `MIGRATIONS_DIR` | no | `migrations` | path used by `cmd/migrate` |

## Commands

| Target | Raw command | Purpose |
|---|---|---|
| `make up` | `docker compose up --build -d` | full stack in Docker |
| `make down` | `docker compose down -v` | stop + drop volumes |
| `make dev` | `go run ./cmd/api` | API natively |
| `make worker` | `go run ./cmd/worker` | worker natively |
| `make migrate` | `go run ./cmd/migrate` | apply migrations natively |
| `make test` | `go test ./...` | run all tests |
| `make tidy` | `go mod tidy` | resolve deps, generate `go.sum` |
