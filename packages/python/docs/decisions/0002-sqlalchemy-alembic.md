# 0002 — SQLAlchemy 2 (async, asyncpg) + Alembic migrations

## Status

Accepted

## Context

Upstream persists data with MikroORM entities (`data/entities.ts` per module) on PostgreSQL, with per-module migrations (`packages/core/src/modules/<module>/migrations/`). The port must keep the exact schema (table/column names, types, defaults) so both stacks can run against the same database, and it needs real migration tooling.

## Decision

- SQLAlchemy 2 with the async engine and asyncpg driver; module entities are declarative classes on a shared `Base` (src/om/shared/db.py).
- Alembic for migrations, configured in `migrations/env.py` to import all registered modules first so `Base.metadata` covers every ported entity and `--autogenerate` works.
- `DATABASE_URL` keeps the upstream `postgres://` scheme; `Settings.sqlalchemy_database_url` normalizes it to `postgresql+asyncpg://` internally so env files stay identical across stacks.

## Consequences

- Autogenerated migrations must be reviewed against the upstream MikroORM migration for the same module — column order and index names can differ; the SQL effect must not.
- One global Alembic version history (migrations/versions/) instead of upstream's per-module folders; migration messages are prefixed with the module id by convention (see AGENTS.md → Data Layer).
- asyncpg means no synchronous session escape hatch; all data access is async.
